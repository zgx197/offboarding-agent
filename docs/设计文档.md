# offboarding-agent 设计文档

## 1. 背景与问题定义

在研发团队中，离职交接通常存在以下问题：

- 交接内容高度依赖个人经验，缺乏统一模板和最低质量标准。
- 交接人常常从“自己熟悉的角度”写文档，导致接手人难以建立完整心智模型。
- 很多风险并不直接体现在代码中，而是隐藏在历史演进、手工流程、外部依赖和默认约定里。
- 交接文档通常只生成一次，后续缺少迭代机制，导致内容快速过期。

`offboarding-agent` 的目标不是简单生成一篇文档，而是提供一套以 `Codex` 为核心分析能力、以本地脚本和状态文件为支撑的交接工作流框架。

## 2. 项目目标

### 2.1 业务目标

- 降低关键模块随人员流动带来的知识断层风险。
- 提高离职交接文档的一致性、可读性和可追溯性。
- 为接手人提供从“快速上手”到“深入维护”的分层材料。
- 将交接从“单次任务”沉淀为“可持续复用的工程能力”。

### 2.2 技术目标

- 支持对指定代码库或子模块进行自动扫描与结构化分析。
- 支持多阶段产物落盘，便于回看、追问、补充和二次生成。
- 支持根据不同目标受众生成不同格式和粒度的输出文档。
- 为未来接入更多项目场景保留扩展点。

## 3. 设计边界

### 3.1 第一版要解决的问题

- 识别交接范围并建立任务上下文。
- 扫描代码库中的关键目录、核心入口、重要配置和现有文档。
- 基于证据生成模块画像、系统关系图描述和风险清单。
- 生成详细交接文档、接手清单和待补问题列表。

### 3.2 第一版暂不解决的问题

- 不强制对所有编程语言都做深度静态分析。
- 不尝试完全替代离职人的人工补充。
- 不实现复杂的 Web UI 或多人在线协作平台。
- 不把所有历史会议记录、聊天记录自动纳入分析范围。

## 4. 总体架构

系统采用“编排层 + 专职 Agent 层 + 工具层 + 状态层 + 输出层”的结构。

### 4.1 编排层

`Orchestrator` 负责组织整体工作流，按顺序或条件触发各阶段任务：

1. 定义交接范围
2. 采集证据
3. 生成系统画像
4. 识别风险与缺口
5. 生成问题清单
6. 输出文档草稿
7. 审核一致性并生成最终材料

### 4.2 专职 Agent 层

建议拆分为以下角色：

- `Scope Agent`
  负责确认本次交接的边界、模块路径、目标受众和交付物类型。
- `Evidence Agent`
  负责扫描代码、文档、配置和历史痕迹，并生成证据清单。
- `Architecture Agent`
  负责梳理模块职责、调用链路、关键入口和依赖关系。
- `Risk Agent`
  负责识别高风险点、隐性依赖、手工步骤和技术债。
- `Interview Agent`
  负责根据现有信息缺口提出需要作者补充的问题。
- `Writer Agent`
  负责生成面向不同受众的输出文档。
- `Reviewer Agent`
  负责检查文档是否存在遗漏、矛盾或证据不足的问题。

### 4.3 工具层

工具层负责处理稳定、可确定执行的机械任务：

- 目录扫描
- 文件索引
- 关键模式检索
- 文档发现
- 变更记录汇总
- 输出模板渲染

工具层的目标是减少大模型在重复性提取工作上的负担，把上下文预算留给分析和写作。

### 4.4 状态层

为了支持断点续跑和多人协作，关键中间结果必须落盘。建议将每次执行结果保存到独立 `run` 目录中。

推荐状态文件示例：

- `task.json`
- `scope.md`
- `evidence.json`
- `module-profiles.json`
- `risks.json`
- `open-questions.md`
- `draft-handover.md`
- `review-report.md`

### 4.5 输出层

第一版建议至少支持四类输出：

- `离职交接总文档`
- `接手优先级清单`
- `风险与技术债清单`
- `管理层摘要`

## 5. 核心数据模型

### 5.1 HandoverTask

用于描述一次完整交接任务。

建议字段：

- `taskId`
- `repoPath`
- `targetPaths`
- `owner`
- `reviewer`
- `audiences`
- `deliverables`
- `status`
- `createdAt`
- `updatedAt`

### 5.2 EvidenceItem

表示一条可追溯的事实证据。

建议字段：

- `id`
- `type`
- `title`
- `path`
- `snippet`
- `summary`
- `tags`
- `sourceStage`

### 5.3 ModuleProfile

表示对单一模块的结构化画像。

建议字段：

- `name`
- `responsibility`
- `entryPoints`
- `coreFlows`
- `dependencies`
- `outputs`
- `operationalNotes`
- `commonPitfalls`

### 5.4 RiskItem

表示一个交接风险或技术债项。

建议字段：

- `title`
- `severity`
- `scope`
- `trigger`
- `impact`
- `mitigation`
- `evidenceIds`

### 5.5 OpenQuestion

表示当前仍需作者或接手人补充确认的问题。

建议字段：

- `question`
- `reason`
- `relatedPaths`
- `priority`

## 6. 典型工作流

### 6.1 初始化阶段

- 输入仓库路径、交接模块和输出目录。
- 读取已有文档、模板和历史产物。
- 生成任务编号并建立 `run` 目录。

### 6.2 证据采集阶段

- 扫描目标目录树。
- 识别入口类、配置文件、注册点和关键脚本。
- 发现已有架构文档、README 和问题记录。
- 对关键检索结果生成摘要并落盘。

### 6.3 分析阶段

- 基于证据抽取模块职责与主流程。
- 识别对外依赖、隐性配置和跨模块耦合点。
- 整理近期热点改动和潜在脆弱区域。

### 6.4 补充阶段

- 输出当前缺失的信息项。
- 引导离职人补充无法从代码中直接得到的知识。
- 记录补充来源，纳入后续文档。

### 6.5 输出阶段

- 生成详细交接文档。
- 生成接手优先级清单和风险清单。
- 生成面向管理者的摘要版内容。
- 执行审校，补充明显遗漏项。

## 8. 首版 MVP 建议

### 8.1 MVP 范围

- 提供基础 `README` 和设计文档。
- 提供一套统一输出模板。
- 提供最小扫描脚本，支持目录树、文档索引和关键类检索。
- 通过 Codex 完成手动触发式的交接文档生成。

### 8.2 MVP 成功标准

- 能针对一个中型模块输出一份结构完整的交接文档。
- 文档中包含明确的入口、职责、风险、依赖和待补问题。
- 关键结论可以追溯到代码、文档或人工补充证据。
- 同一工作流可以复用到第二个模块，而不需要重写大段提示词。

## 9. 风险与挑战

- 仅依赖代码扫描无法获得全部隐性知识，因此必须保留人工补充环节。
- 如果状态文件设计不清晰，后续多阶段协作会快速失控。
- 如果输出模板过于僵硬，会降低文档的实际可用性。
- 如果所有分析都放在单一 Agent 中，容易出现上下文膨胀和结果不可追溯的问题。

## 10. 后续演进建议

- 增加面向不同技术栈的扫描器适配层。
- 增加文档质量评分与缺失项检查。
- 增加面向新人 onboarding 的二次产物生成能力。
- 增加历史 `run` 对比功能，用于跟踪模块知识沉淀是否完善。

## 11. 结论

`offboarding-agent` 不应被设计成一个一次性写文档的“大 prompt”，而应被设计成一个可拆分、可追踪、可扩展的交接工作流框架。`Codex` 适合作为该框架中的核心推理与写作引擎，而证据采集、状态管理和输出模板应由本地工程资产承担。这样的组合方式更稳定，也更适合在真实团队环境中长期演进。
