# offboarding-agent

一个基于 Codex 的工程离职交接 Agent 框架，用于自动梳理代码库结构、沉淀模块知识、识别风险与技术债，并生成可复用的交接文档、接手清单和知识传承材料。

## 项目目标

`offboarding-agent` 旨在解决研发团队在人员离职、模块转交、项目接手和知识传承过程中的常见问题：

- 关键知识高度依赖个人经验，缺乏结构化沉淀。
- 交接文档质量不稳定，常常只覆盖表层信息。
- 接手人不知道应该先看哪里，也不知道哪些点风险最高。
- 很多隐性约束存在于代码历史、手工流程和口头经验中，难以被系统化保留。

本项目希望通过 `Codex + 本地脚本 + 模板 + 状态文件` 的组合方式，将离职交接从“一次性文档输出”升级为“可重复执行、可追踪、可迭代补全”的工程工作流。

## 核心能力

- 自动扫描目标代码库的目录结构、关键入口、配置项和文档资产。
- 以证据驱动的方式抽取模块职责、调用链路、关键依赖和常见操作流程。
- 识别高风险模块、隐性知识、技术债和待补信息缺口。
- 面向不同受众输出多种交付物，例如详细交接文档、接手清单、风险清单和管理摘要。
- 支持多轮迭代补全，将“代码可见信息”和“作者补充信息”统一沉淀。

## 设计原则

- `证据优先`：先采集事实，再输出结论，避免生成看似完整但缺乏依据的文档。
- `流程拆分`：将范围识别、信息采集、架构梳理、风险分析、追问补充和文档写作拆成独立阶段。
- `状态落盘`：中间结果采用 Markdown 或 JSON 持久化，支持断点续跑和多人协作。
- `人机协同`：允许作者对自动生成的初稿进行补充，而不是强求全自动。
- `可迁移`：框架设计尽量不与单一项目强绑定，支持扩展到不同代码库和业务域。

## 首版规划

第一版将聚焦于一个最小可用的离职交接工作流：

1. 输入交接范围、目标目录和输出要求。
2. 扫描代码、文档和关键配置，生成证据清单。
3. 根据证据生成系统画像、风险清单和待补问题。
4. 补充作者经验后输出交接文档初稿。
5. 对输出结果进行一致性检查，并生成最终交付材料。

## 当前 MVP 实现

当前仓库已经提供第一版本地命令行工作流，用于在**不修改目标代码库内容**的前提下生成初版交接材料。

实现入口：

- `scripts/offboarding_agent.py`

当前命令：

```bash
python scripts/offboarding_agent.py run --repo <目标仓库路径> --target <目标模块路径>
```

常用参数：

- `--repo`：被交接的目标仓库路径
- `--target`：需要交接的模块路径，可重复传入多个
- `--owner`：交接人
- `--audience`：输出材料的主要受众
- `--reviewer`：审阅人
- `--task-id`：可选，自定义本次任务编号

示例：

```powershell
python scripts/offboarding_agent.py run `
  --repo D:\work\payment-service `
  --target src\billing `
  --target scripts `
  --owner Alice `
  --audience 接手开发 `
  --task-id billing-handover
```

输出约束：

- 扫描过程只读取 `--repo` 指向的目标仓库
- 所有产物只写入当前工具仓库的 `runs/<taskId>/`
- 不会在目标仓库中创建、修改或删除任何文件

当前生成产物：

- `task.json`
- `evidence.json`
- `scan-summary.md`
- `handover.md`
- `open-questions.md`
