# understanding/ 模板说明

这组模板用于生成 `runs/<taskId>/understanding/` 目录下的中间理解产物。
它们的定位是“内部生成层”，主要服务 Agent 建模、证据回溯和最终 handover 整合生成，不是默认交付给接手人的阅读入口。

设计原则：

- 先服务 Agent 建立稳定心智模型，再服务后续文档写作。
- 先写结论，再写支撑，不复制长篇原始证据。
- 每条重要结论都应回指 `evidence.json` 中的证据 id。
- 明确区分 `事实`、`推断` 和 `未知`。
- 所有路径统一使用仓库相对路径。
- 面向接手人的最终产物应尽量收敛为单文档 handover；`understanding/` 负责提供可整合的结构化中间层。

建议约定：

- Markdown 模板统一使用 YAML front matter。
- JSON 模板统一使用 `meta` + `items` 的顶层结构。
- 后续脚本若要自动填充，可直接复用这些模板文件名。

新增能力说明：

- `14-code-evidence-registry.json`
  - 用于把“高层结论”继续下钻到“可定位的代码锚点”。
  - 每条记录应明确：
    - 代码路径
    - 符号
    - 行号
    - 这段代码证明了什么
    - 它支持哪些结论 / 契约 / 风险判断
  - 目标不是做完整代码索引，而是注册“不能跳过的关键代码证据”。

- `15-change-impact-matrix.md`
  - 用于回答“如果改这里，会影响哪些层、哪些产物、哪些外部消费者”。
  - 每一行都应面向真实改动面，而不是抽象模块名。
  - 理想输出应让 Agent 或接手人能直接据此制定回归清单。

推荐使用顺序：

1. 先用 `00-13` 建立整体理解。
2. 再用 `14-code-evidence-registry.json` 把核心结论绑定到具体代码锚点。
3. 最后用 `15-change-impact-matrix.md` 形成“改动 -> 影响 -> 回归”的推理闭环。
